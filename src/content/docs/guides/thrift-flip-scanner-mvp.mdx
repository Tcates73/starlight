# Thrift Flip Scanner

A mobile-first “thrift arbitrage operating system” that gives a **Buy / Maybe / Pass** recommendation in under 10 seconds while a reseller is standing in-store.

## 1) Full Product Spec

### One-line value proposition
**Scan thrift finds and know the real flip before you buy.**

### Primary users
- Part-time resellers
- Vintage & clothing flippers
- Shoe and bag arbitrage buyers
- Small-electronics pickers
- Home-goods resellers

### Jobs-to-be-done
1. Identify item quickly.
2. Find reliable sold comps (not active listings).
3. Estimate real net after fees + shipping + packaging + COGS.
4. Decide instantly whether to buy.

### MVP scope (v1)
- Categories: clothing, shoes, bags, small electronics, home goods.
- Marketplace comps: **eBay sold listings first** (expandable adapters for others).
- Recommendation engine with customizable thresholds.
- Manual correction of all key fields.
- Guest mode + optional magic link auth.
- Scan history + save/recheck/share/list-later actions.

### Out of scope for v1
- Live carrier API rate shopping.
- Barcode-native mode (planned).
- Team collaboration.
- Crosslisting automation.

### Key product principles
- **Decision speed over data density.**
- **Explicit uncertainty handling** (sparse comps, wide ranges, low confidence).
- **One-hand usability** with large targets and minimal navigation depth.
- **Instant feedback states** during processing.

### Success metrics
- Time to first recommendation (TTFR): target p50 < 8s, p90 < 12s.
- Scan save rate: target > 35%.
- Free → Pro conversion: target > 6% by day 30.
- “Buy” to listed inventory conversion (self-reported): target > 45%.
- Decision confidence score (in-app survey): target avg ≥ 4.2/5.

## 2) App Architecture

### Stack
- Frontend: Next.js (App Router) + React + Tailwind, PWA-ready.
- Backend: Supabase (Postgres + Auth + Storage + Edge Functions).
- AI pipeline:
  - Vision model adapter (classification + visual attributes)
  - OCR adapter (label/tag/model extraction)
  - LLM query builder (vision+OCR → sold-comp query terms)
- Marketplace adapter layer (start with eBay sold).

### High-level system diagram
1. User captures photo.
2. `/api/scan` creates scan job and returns `scan_id` immediately.
3. Async pipeline executes:
   - Vision inference
   - OCR extraction
   - Query synthesis
   - Comp fetch + normalization
   - Pricing, fee, shipping, profit math
   - Recommendation engine
4. Client subscribes to scan status updates and progressively renders.
5. Final result persisted to `scans`, `scan_results`, and `comps` tables.

### Service boundaries (swappable)
- `VisionProvider` interface
- `OCRProvider` interface
- `CompProvider` interface
- `FeeCalculator` module
- `ShippingEstimator` module
- `RecommendationEngine` module

### Performance design
- Fast response pattern:
  - optimistic UI with processing milestones
  - partial result hydration (identity first, comps second, decision final)
- Cache normalized sold comps by hash(query + category + size band) for 15 minutes.
- Background retries for transient comp API failures.

## 8) Core Recommendation Algorithm (Pseudocode)

```pseudo
function recommend(scan, userSettings):
  identityConf = scan.identity.confidence
  compCount = scan.summary.comp_count
  compConf = scan.summary.comp_confidence
  rangeRatio = scan.summary.price_range_ratio   // high/low

  minNet = thresholdForCategory(userSettings, scan.identity.category, "min_net_profit", default=15)
  minRoi = thresholdForCategory(userSettings, scan.identity.category, "min_roi_percent", default=100)
  minConf = userSettings.min_confidence

  net = scan.profit.net_profit
  roi = scan.profit.roi_percent

  score = 0
  reasons = []
  risks = []

  if identityConf >= minConf: score += 20
  else:
    score -= 20
    risks.push("Low identification confidence")

  if compCount >= 8 and compConf >= 0.75:
    score += 25
    reasons.push("Strong sold comp depth")
  else if compCount >= 4:
    score += 10
    risks.push("Comps are somewhat sparse")
  else:
    score -= 20
    risks.push("Very sparse sold comps")

  if rangeRatio >= 2.5:
    score -= 10
    risks.push("Wide sold range; condition-sensitive")

  if net >= minNet:
    score += 25
    reasons.push("Net profit beats threshold")
  else:
    score -= 25

  if roi >= minRoi:
    score += 20
    reasons.push("ROI beats threshold")
  else:
    score -= 15

  if compCount < 4 and net >= minNet and roi >= minRoi:
    score -= 8  // uncertainty downgrade guardrail

  if score >= 55:
    rec = "buy"
  else if score >= 30:
    rec = "maybe"
  else:
    rec = "pass"

  return { rec, score, reasons, risks }
```

## 10) Polished First Version UI (Component Spec)

### Visual system
- Background: near-black `#0B1020` with elevated cards.
- Decision colors:
  - Buy: `#16A34A`
  - Maybe: `#F59E0B`
  - Pass: `#DC2626`
- Typography: large numeric emphasis for net and ROI.
- Button height: 52px minimum, radius 14px.

### Results card hierarchy
1. Decision badge + confidence pulse.
2. “Expected Net” large number.
3. Resale range + best marketplace chips.
4. Fees/shipping breakdown mini-table.
5. Reason bullets and risk flags.
6. Sticky action row.

### UX micro-interactions
- Tap-to-expand reasoning details.
- Haptic-style visual pulse on recommendation reveal.
- Inline recompute animation when user edits costs.

### Empty/error states
- No comps: “Couldn’t find enough sold comps yet — try recheck or broaden item details.”
- Low confidence: “We’re unsure on exact match. Edit brand/model for better comps.”
- Network fail: local save + retry queue.

### Accessibility
- Contrast ratio ≥ 4.5:1.
- VoiceOver labels for all action buttons.
- Dynamic text support for key metrics.

This UI direction keeps the app fast, decisive, and field-ready for one-handed in-store use.
